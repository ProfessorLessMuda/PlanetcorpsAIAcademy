## Key Concepts

### 5.1 Conversation Context Management

The context window is the model's only memory. Every fact, instruction, and tool result exists only because it is present in the current context window — the model has no persistent state between API calls and no ability to recall information from previous calls unless that information was included in the messages array. This fundamental constraint means that managing what is in the context window, where it is positioned, and how it is represented is not a minor optimization concern — it is an architectural responsibility that directly determines system reliability. A system that ignores context management will work on simple tasks and fail unpredictably on complex ones, often in ways that are difficult to diagnose because the model will confidently produce outputs based on incomplete or degraded context rather than signaling that something is missing.

Progressive summarization is a common strategy for managing long conversations: as the conversation grows, older messages are summarized to free up token space. The technique is sound in principle but dangerous in practice if applied carelessly. The core risk is information loss during summarization. When you condense a 20-turn conversation into a paragraph-length summary, you are making editorial decisions about what matters and what does not. Numerical values are particularly vulnerable — a summary might say "the customer discussed pricing" when the actual conversation established that the customer was quoted $4,750 for a three-year contract with a 15% early termination fee. Those specific numbers are critical to any subsequent actions the agent takes, but they are exactly the kind of detail that gets dropped during summarization because they appear less "important" than the narrative flow. Dates, deadlines, exact product names, version numbers, account identifiers, and precise customer expectations all share this vulnerability. A progressive summarization strategy must explicitly extract and pin these critical values before compressing the surrounding narrative.

The practical implementation of context pinning works as follows. Before summarizing a conversation segment, a preprocessing step scans for and extracts structured data points: monetary amounts, dates, identifiers, specific commitments, and quoted requirements. These extracted values are formatted into a structured block — often a simple key-value listing or a short JSON-like section — and placed at the beginning of the summarized segment. The narrative summary follows, providing the conversational flow and reasoning. This ensures that even as the narrative is compressed, the precise data points survive intact. The pinned values section should be treated as immutable during subsequent summarization rounds — it accumulates rather than being re-summarized. This creates a growing "facts" section at the top of the context and a compressed narrative section that continues to be summarized as needed.

The "lost in the middle" effect is a well-documented phenomenon in large language models: information placed in the middle of a long context window receives less attention than information at the beginning or end. This is not a theoretical concern — it has measurable impact on task performance. In a long conversation, the system prompt occupies the beginning, and the most recent user message and tool results occupy the end. Everything in between — which may include critical decisions, established constraints, and agreed-upon parameters from earlier in the conversation — sits in the attention trough. The practical consequence is that the model may "forget" or underweight constraints that were clearly established earlier in the conversation, not because the information is absent from the context, but because the model's attention mechanism does not weight it sufficiently.

Mitigation strategies for the lost-in-the-middle effect include re-stating critical constraints in the most recent user message or in a system prompt that is refreshed periodically, using structured summaries that surface key data points in a scannable format rather than burying them in narrative paragraphs, and designing the conversation structure so that the most important information appears near the boundaries (beginning or end) of the context window rather than exclusively in the middle. Some architectures address this by including a "current state" block that is regenerated and placed at the end of the context before each model call, containing the essential facts and constraints that must govern the model's next response.

Tool results are a major and often underestimated contributor to context bloat. Each tool call generates a result that is appended to the conversation history, and many tool results are verbose — a database query might return dozens of rows, a file read might include hundreds of lines, and an API response might contain deeply nested JSON with many fields the model does not need. Over multiple tool calls, these results accumulate and can consume a disproportionate share of the context window, pushing out the earlier conversation context that contains the user's original requirements and the reasoning that led to the current approach. This is insidious because the context window appears full and active — it just contains the wrong information. The model has detailed tool outputs from its recent actions but has lost the strategic context about why it is taking those actions and what constraints apply.

Strategies for managing tool result accumulation include truncating or filtering tool results before appending them (removing irrelevant fields from API responses, limiting database results to the most relevant rows), summarizing verbose tool results into a concise finding before appending, and using subagent delegation to isolate verbose exploration (the subagent consumes the full tool results in its own context, and only its synthesized conclusion is returned to the coordinator's context). The choice among these strategies depends on the application: truncation works well when you know in advance which fields matter, summarization is better when relevance depends on context, and subagent delegation is appropriate when the exploration itself is complex and multi-step.

A common exam mistake in this area is recommending aggressive summarization without acknowledging the information loss risk. The correct approach always involves a strategy for preserving critical data points through the summarization process — not treating summarization as a lossless compression technique. Another common mistake is assuming the model will "remember" something because it was mentioned earlier in the conversation. If the context is long enough, and the information is positioned in the middle, the model may not attend to it sufficiently. Explicit repetition and structured formatting are not redundant — they are reliability mechanisms.

### 5.2 Escalation and Ambiguity Resolution

Escalation — the transfer of a task from an automated agent to a human operator or to a higher-authority system — is one of the most important reliability mechanisms in any production agent deployment. A system that cannot escalate effectively is a system that will eventually cause harm: it will make a decision it is not qualified to make, take an action it should not take, or leave a user stuck in a loop with no path to resolution. Designing effective escalation is not about admitting the model's limitations after the fact — it is about building a system architecture that recognizes those limitations by design and routes around them before they cause problems.

Reliable escalation triggers are conditions that can be detected deterministically or with high confidence and that genuinely indicate the agent should not continue autonomously. There are three categories that consistently produce reliable escalation decisions. First, the customer or user explicitly requests a human. This is the most reliable trigger because it requires no inference — the user has directly stated their preference, and honoring it is both correct and important for trust. Even if the agent believes it can resolve the issue, overriding an explicit escalation request damages trust and should never be done. Second, a policy exception or gap is detected. This occurs when the agent encounters a situation that falls outside its authorized decision space — a refund amount that exceeds its threshold, a regulatory question it is not qualified to answer, a complaint that triggers a specific escalation policy. These triggers are reliable because they are defined by the business logic, not by the model's judgment. Third, the agent is unable to make progress after a reasonable number of attempts. If the agent has tried multiple approaches and is cycling without resolution — repeating the same tool calls, generating the same clarifying questions, or hitting the same errors — it should escalate rather than continuing to waste the user's time. The definition of "reasonable number" should be configured, not left to the model's discretion.

Unreliable escalation triggers are conditions that seem intuitively useful but perform poorly in practice. Sentiment-based escalation — escalating when the model detects that the user is frustrated or angry — is unreliable because large language models are poor at real-time sentiment analysis in conversational contexts. Sentiment is nuanced, culturally dependent, and easily misread. A user who writes in short, direct sentences may be perceived as angry when they are simply being efficient. A user who uses polite language may be deeply frustrated but masking it. Sarcasm, irony, and understatement are particularly difficult for models to detect. Building escalation logic around sentiment detection creates a system that escalates unpredictably — sometimes escalating calm users who happen to use direct language, and sometimes failing to escalate genuinely frustrated users who maintain a polite tone. The exam consistently treats sentiment-based escalation as a wrong answer.

Self-reported confidence scores are another unreliable escalation mechanism. The idea seems logical: have the model assess its own confidence in its response, and escalate when confidence is low. The problem is that models do not have reliable introspective access to their own uncertainty. A model that is wrong about a factual claim often reports high confidence in that claim — this is the core of the hallucination problem. Conversely, a model might report low confidence on a response that is entirely correct because the phrasing of the prompt triggered hedging behavior. Self-reported confidence is a language generation behavior, not a measurement of actual reliability. Using it as an escalation trigger means you are building your safety mechanism on exactly the capability (accurate self-assessment) that the model lacks. Calibrated confidence requires external validation infrastructure (covered in Task 5.5), not model self-reporting.

Ambiguity resolution is closely related to escalation but addresses a different scenario: the agent has enough information to identify multiple possible interpretations but not enough to determine which is correct. The critical design principle here is to always clarify with the user rather than guessing. When a customer asks about "my order" and has three open orders, the agent should not apply heuristics (most recent, highest value, most likely to have issues) to guess which one — it should ask the customer which order they mean. When a user asks to "update the configuration" and there are configuration files in multiple locations, the agent should list the options and ask for clarification rather than modifying the most likely one. Guessing introduces silent errors: the agent proceeds confidently with the wrong interpretation, and the user may not realize the mistake until the consequences become apparent. Asking for clarification has a small cost in conversational efficiency but a large benefit in accuracy and trust.

The distinction between immediate escalation and resolution-first escalation matters for exam questions. Immediate escalation is appropriate for safety-critical situations (a user reports a medical emergency through a support channel), legal situations (a user threatens litigation), regulatory compliance situations (a request that may violate data protection rules), and explicit user requests for a human. In these cases, attempting resolution first would be inappropriate or risky — the agent should escalate immediately with full context. Resolution-first escalation is appropriate for routine issues where the agent has a reasonable chance of success: a billing question, a product inquiry, a standard troubleshooting flow. In these cases, the agent should attempt to resolve the issue, and escalate only if it cannot. The key is that the choice between these patterns is determined by the nature of the situation, not by the model's confidence in its ability to handle it.

### 5.3 Error Propagation in Multi-Agent Systems

When a subagent encounters an error in a multi-agent system, how it communicates that error back to the coordinator determines whether the system can recover gracefully or fails opaquely. This is not a minor implementation detail — it is one of the most consequential design decisions in multi-agent architecture. A coordinator that receives a well-structured error report can make an informed decision: retry with different parameters, delegate to an alternative subagent, degrade gracefully by working with partial results, or escalate to a human with a clear explanation of what went wrong. A coordinator that receives a generic error status can do none of these things — it can only report failure or blindly retry without addressing the underlying cause.

Structured error context means the subagent returns a response that includes, at minimum: what operation failed (the specific tool call, query, or action that did not succeed), why it failed (the error type, error message, or condition that caused the failure), whether the failure is retryable (a timeout is retryable; an authentication failure is not; a rate limit is retryable after a delay), and what alternatives exist (if the primary approach failed, are there fallback data sources, alternative APIs, or different strategies the coordinator could try). This structured information transforms an opaque failure into an actionable situation report. The coordinator can inspect the error context and make a reasoned decision rather than applying a one-size-fits-all error handling strategy.

Consider the concrete difference between these two error responses from a database subagent. The first returns: `{ "status": "error" }`. The coordinator knows that something went wrong but nothing else. It cannot distinguish between the database being unreachable (infrastructure problem, possibly temporary), the query being malformed (logic error, retrying will not help), the query returning no results (not actually an error — just an empty result set), or the subagent lacking the necessary credentials (configuration problem, requires human intervention). All four situations require different handling, but the generic error status hides the distinction.

The second response returns: `{ "status": "error", "error_type": "connection_timeout", "operation": "query_customer_orders", "parameters": { "customer_id": "C-4821" }, "retryable": true, "retry_after_seconds": 5, "message": "Database connection timed out after 30s", "alternatives": ["cache_lookup", "fallback_read_replica"] }`. Now the coordinator knows exactly what happened, knows the failure is temporary, knows it should wait before retrying, and has two alternative approaches it can try if the retry fails. This is the difference between a system that degrades gracefully and one that falls over at the first error.

The most critical distinction in error propagation — and one that appears repeatedly on the exam — is between an access failure and a valid empty result. An access failure means the subagent could not reach the data source: the API was down, the network connection failed, the authentication was rejected, or the request timed out. A valid empty result means the subagent successfully queried the data source and the data source returned no matching records. These two outcomes have completely different implications. If a customer order lookup returns an access failure, the coordinator should retry or use a fallback — the customer might well have orders that simply could not be retrieved. If the lookup returns a valid empty result, the coordinator should inform the customer that no orders were found. Treating an access failure as "no results found" is a category error that leads to telling customers they have no orders when in fact the system just could not look them up. Treating a valid empty result as an error leads to unnecessary retries and confusion.

Two major anti-patterns dominate error handling in multi-agent systems. The first is silent error suppression: the subagent encounters an error, handles it internally (perhaps by returning a default value or an empty result set), and reports success to the coordinator. The coordinator has no idea that something went wrong and proceeds to build its output on top of incomplete or substituted data. This is particularly dangerous when the subagent returns plausible-looking default data — the coordinator, and ultimately the user, have no way to know that the information is fabricated rather than retrieved. Silent suppression is often introduced by well-meaning developers who want the system to be "resilient," but resilience requires visibility into failures, not hiding them.

The second anti-pattern is terminating the entire workflow on any error. In this pattern, any subagent failure causes the coordinator to abort the entire task and report failure to the user. This is overly conservative and throws away all the work that other subagents may have completed successfully. If a three-subagent system has two subagents complete successfully and the third encounters a retryable error, aborting the entire workflow wastes the successful results and forces the user to start over. Graceful degradation is almost always preferable: the coordinator should attempt recovery for the failed subagent, and if recovery is not possible, it should deliver the partial results with a clear indication of what is missing and why. A research report that covers three of four requested topics, with a note that the fourth could not be retrieved due to a temporary service outage, is far more useful than no report at all.

Implementing effective error propagation requires a standardized error schema that all subagents use. This schema should be defined at the system level, not left to each subagent to invent independently. When subagents use inconsistent error formats, the coordinator must implement custom parsing logic for each one, which is fragile and error-prone. A standardized schema ensures that the coordinator can apply consistent error handling logic regardless of which subagent reported the error. The schema should include fields for error severity (warning, error, fatal), error category (network, authentication, validation, resource_not_found, rate_limit), retryability, and structured context specific to the error category.

### 5.4 Large Codebase Context Management

Exploring a large codebase is one of the most context-intensive tasks a model can perform. Each file read, each search result, and each tool output consumes context window space, and a meaningful exploration of a large codebase might involve reading dozens of files across multiple directories. The fundamental challenge is context degradation: as the agent explores more files, the context window fills with file contents and tool results, and earlier findings — the architectural insights, the dependency relationships, the patterns discovered in the first few files — fade from the model's effective attention. This is the lost-in-the-middle problem applied to code exploration, and it is particularly acute because code exploration is inherently sequential and cumulative. Understanding file 20 often depends on insights gained from files 1 through 19, but by the time the agent reaches file 20, those insights may no longer be receiving sufficient attention.

The scratchpad file pattern is the primary mitigation strategy for context degradation during extended exploration. The idea is simple but powerful: as the agent explores the codebase, it periodically writes its findings to an external file — a scratchpad or working notes file. This file serves as external memory that persists beyond the context window's effective attention span. When the agent discovers a key architectural pattern in an early file, it writes a note to the scratchpad. When it finds a dependency relationship, it adds it to the scratchpad. When it identifies a potential issue, it records it. Later in the exploration, if the agent needs to recall those earlier findings, it can re-read the scratchpad file rather than relying on the degraded attention to those earlier conversation turns.

The scratchpad is effective because it transforms the agent's exploration from a purely sequential conversation (where all findings compete for attention in the context window) into a structured knowledge accumulation process (where findings are externalized and can be selectively re-loaded). The scratchpad file should be structured rather than free-form — using sections, headings, or key-value formats rather than narrative paragraphs — so that re-reading it is efficient and the agent can quickly locate specific earlier findings without processing a wall of text. Practical implementations often include sections like "Architecture Overview," "Key Dependencies," "Open Questions," and "Potential Issues," which the agent updates incrementally as the exploration progresses.

Subagent delegation is the second key strategy for large codebase management. Instead of having a single agent explore the entire codebase in one long session — accumulating verbose tool results in a single context window — the coordinator delegates specific exploration tasks to subagents. For example: "Examine the authentication module and return a summary of its architecture and key dependencies," or "Search for all uses of the DatabaseClient class and document the patterns you find." Each subagent conducts its exploration in its own isolated context, consumes all the verbose file contents and search results within that context, and returns only a synthesized summary to the coordinator. This keeps the coordinator's context clean and focused — it receives the insights without the raw data that produced them.

The combination of scratchpad files and subagent delegation creates a powerful exploration architecture. The coordinator maintains the high-level exploration plan and the scratchpad. It delegates specific investigation tasks to subagents, receives their synthesized findings, and writes those findings to the scratchpad. It then reviews the accumulated findings in the scratchpad to determine what to investigate next. This approach scales to very large codebases because the coordinator's context never fills with raw file contents — it works with synthesized summaries and a structured scratchpad — while the subagents handle the verbose exploration in isolated contexts that are discarded after their findings are captured.

Structured state persistence extends the scratchpad concept to support crash recovery and session continuity. In long-running exploration tasks, the agent may encounter context window limits, session timeouts, or other interruptions. If the exploration state is stored only in the conversation context, any interruption means starting over. By periodically saving the full exploration state to files — not just findings but also the exploration plan, the list of files already examined, the open questions, and the next steps — the agent can resume from where it left off after an interruption. This is particularly important for tasks that take hours or span multiple sessions. The saved state files serve as both a recovery mechanism and a progress record that can be reviewed by the user.

The common exam mistake in this area is describing an exploration approach that reads many files in a single session without any mechanism for externalizing findings. The question will describe a scenario where the agent needs to analyze a large codebase, and the wrong answer will involve a single agent reading file after file in a long conversation. The correct answer will involve either scratchpad files (for findings persistence), subagent delegation (for isolating verbose exploration), or both. The key concept to demonstrate is awareness that the context window is a finite resource and that extended exploration requires explicit strategies for managing it.

### 5.5 Human Review Workflows and Confidence Calibration

When an automated system processes documents, extracts data, or makes decisions at scale, measuring the quality of its output is not optional — it is a core architectural requirement. Human review workflows exist to answer a deceptively simple question: how well is the system actually performing? The challenge is that naive approaches to measuring performance can produce misleading results that create false confidence and hide systematic failures. Designing a review workflow that produces trustworthy quality measurements requires understanding sampling strategies, accuracy metrics, and the limitations of model self-assessment.

Aggregate accuracy metrics — a single number like "95% accuracy" — are the most common and most dangerous way to measure system performance. They are dangerous because they can mask severe problems with specific categories, document types, or fields. Consider a document processing system that handles invoices (80% of volume) and contracts (20% of volume). Invoices are relatively standardized and the system achieves 99% accuracy on them. Contracts are complex and varied, and the system achieves 80% accuracy on them. The aggregate accuracy is 0.8 * 99% + 0.2 * 80% = 95.2%. This overall number looks excellent, but one in five contracts is being processed incorrectly — a potentially serious problem that the aggregate metric completely obscures. The business impact may be concentrated in the contracts because contract errors are more consequential than invoice errors, making the 95.2% aggregate not just misleading but inversely correlated with actual risk.

The same masking effect occurs at the field level. A data extraction system might achieve 98% accuracy overall, but when you break it down by field, names are extracted at 99.5% accuracy, addresses at 97%, dates at 92%, and monetary amounts at 85%. That 85% on monetary amounts — the most financially consequential field — is hidden by the strong performance on the high-volume, easier fields. If the business is relying on the aggregate 98% to make decisions about whether to trust the system's financial extractions, they are operating under a dangerous illusion. Accuracy must be measured per document type and per field, not just in aggregate.

Stratified random sampling is the measurement technique that addresses the aggregate masking problem. Instead of randomly sampling from the full output (which will naturally oversample high-volume categories), stratified sampling ensures that each category is represented in the review sample proportionally to its importance, not its volume. If you have four document types, you might sample 50 documents from each type for review, regardless of whether one type represents 5% of total volume and another represents 60%. This ensures that you measure accuracy within each category with sufficient statistical confidence to detect problems. The review results are then reported per category, not as a single aggregate — giving the business a clear picture of where the system performs well and where it needs improvement.

The sampling strategy should also account for known difficulty dimensions. If certain fields are more error-prone, the sampling should oversample documents where those fields are present or complex. If certain source formats (handwritten versus printed, scanned versus digital) affect accuracy, the sampling should ensure representation across source formats. The goal is to design a sampling strategy that is most likely to find problems, not one that confirms the system works well on easy cases. This is the opposite of how many systems are evaluated — often, the easiest examples are tested first because they are most convenient, producing artificially high accuracy numbers that collapse when the system encounters harder cases in production.

Field-level confidence scores — the model's self-assessment of how confident it is in each extracted value — seem like a natural mechanism for routing uncertain extractions to human review. If the model reports low confidence on a specific field, that extraction is flagged for human verification; high-confidence extractions are accepted automatically. This is an appealing design, but it requires careful calibration to be trustworthy. Out of the box, model confidence scores are not well-calibrated: when the model says it is 90% confident, the actual accuracy may be 95% or 70%, depending on the field type, document type, and extraction complexity. Uncalibrated confidence scores lead to either over-routing (too many extractions sent to humans, defeating the purpose of automation) or under-routing (incorrect extractions passing through because the model was overconfident).

Calibrating confidence scores requires a labeled validation set — a set of documents with known correct values for each field. The system processes these documents, records both its extracted values and its confidence scores, and then compares the extractions to the known correct values. This produces a calibration curve: for each confidence level, what is the actual accuracy? If the model says 90% confidence and the actual accuracy at that level is 82%, the confidence threshold for human review needs to be set higher than 90% to catch errors effectively. This calibration must be performed per field and per document type, because the relationship between reported confidence and actual accuracy varies across these dimensions. A confidence score of 85% on a name field might correspond to 95% actual accuracy, while 85% on a date field might correspond to only 75% actual accuracy.

The common exam mistake in this domain is accepting aggregate accuracy metrics at face value, proposing model self-reported confidence as a reliable quality mechanism without calibration, or designing review workflows that sample randomly rather than strategically. The correct approach always involves stratified measurement, per-category and per-field accuracy reporting, and externally calibrated confidence scores rather than raw model self-assessment.

### 5.6 Information Provenance

When an agent synthesizes information from multiple sources — combining findings from web searches, document analysis, database queries, and other data retrieval operations — the resulting summary is more useful and trustworthy only if each claim can be traced back to its source. Information provenance is the practice of maintaining this traceability: knowing which source said what, when the source was created or last updated, and how reliable the source is. Without provenance, a synthesized report is a collection of unsourced assertions that cannot be verified, challenged, or updated when the underlying sources change.

Source attribution is lost by default during summarization. When a model reads three articles and produces a summary paragraph, the paragraph weaves together information from all three sources into a coherent narrative, but the connection between specific claims and their sources is dissolved. A reader of the summary cannot determine whether a particular statistic came from source A or source B, whether a particular conclusion is supported by one source or all three, or whether a particular claim was a direct finding from a source or an inference the model made by combining information across sources. This loss of attribution is not a model failure — it is the natural result of synthesis. Preserving attribution requires explicit architectural mechanisms; it does not happen automatically.

The structured claim-source mapping pattern addresses attribution loss by requiring the agent to maintain an explicit mapping between claims and sources throughout the synthesis process. Instead of producing a flowing narrative, the agent produces a structured output where each claim or fact is paired with its source. This can take many forms: footnote-style citations, inline source references, a table mapping claims to sources, or a JSON structure where each claim object includes a source field. The specific format matters less than the discipline of maintaining the mapping. The mapping must be created during the synthesis process, not reconstructed afterward — asking a model to retroactively attribute sources to a summary it already generated produces unreliable results because the model may confabulate attributions.

When sources conflict — for example, when two credible sources provide different statistics for the same metric, or when sources reach different conclusions about the same question — the agent must surface the conflict rather than silently resolving it. Silently choosing one source's value over another's introduces bias and removes the reader's ability to evaluate the competing claims. The conflict itself is valuable information: it signals that the data point is uncertain, that more investigation may be needed, or that the difference reflects a legitimate disagreement that the reader should understand. A synthesis that says "Source A reports the market size as $4.2B (2024 data), while Source B reports $3.8B (2023 data)" is more honest and more useful than one that simply states "The market size is $4.2B."

Temporal data management is a specific aspect of provenance that is easy to overlook but critically important. Information has a shelf life that varies by domain. Financial market data may be stale within hours. Population statistics are typically annual. Legal precedents may remain relevant for decades. When an agent synthesizes information from multiple sources, each source has a publication or collection date, and these dates determine how current and relevant the information is. A synthesis that combines a 2020 statistic with a 2024 analysis without noting the temporal gap is misleading — the 2020 data may no longer reflect current reality, and the 2024 analysis may be based on more recent data that supersedes the 2020 figure. Provenance tracking must include temporal metadata: when was this information published, when was it collected, and is it still likely to be current?

Practical implementation of provenance in multi-agent systems requires each subagent to return structured source metadata alongside its findings. When a web search subagent returns results, each result should include the source URL, the publication date, the author or organization, and any relevant credibility indicators. When a document analysis subagent extracts information, each extracted fact should be tagged with the document it came from and the specific section or page. The coordinator then preserves these source tags through the synthesis process, producing a final output where every claim can be traced back through the subagent that found it to the original source that asserted it. This creates a provenance chain — an unbroken link from the final claim through the intermediary processing steps to the original source.

The common exam mistake regarding provenance is designing a synthesis workflow that produces clean, readable narrative outputs without any mechanism for tracking which source contributed which information. The question will typically present a scenario where an agent must research a topic using multiple sources and produce a report. The wrong answer produces a well-written report with no source tracking. The correct answer produces a report with claim-source mappings, surfaces source conflicts rather than hiding them, and includes temporal metadata for assessing data freshness.

## Architecture Patterns

### Context Pinning Pattern
Before any summarization of conversation history occurs, a preprocessing step extracts critical data points — monetary values, dates, identifiers, exact commitments, version numbers, account numbers — and formats them as a structured "pinned facts" block. This block is placed at the top of the summarized content and is excluded from future summarization rounds. It accumulates as the conversation progresses, creating a growing record of precise data that survives context compression. The narrative summary is then applied only to the conversational flow, not to the pinned data. This separates precise facts from narrative context and ensures the facts that matter most for accurate downstream actions are never lost.

### Structured Escalation Pattern
Escalation decisions are driven by deterministic triggers defined in the system's business logic, not by the model's self-assessment. Three trigger categories are defined explicitly: user-requested (the user asks for a human), policy-triggered (the situation falls outside the agent's authorized decision space), and progress-based (the agent has failed to advance toward resolution after a configured number of attempts). Each trigger fires unconditionally when its condition is met. The escalation payload includes a structured handoff document: conversation summary, actions taken and their results, the specific trigger that caused the escalation, and relevant data the human agent will need. Sentiment-based and confidence-based triggers are explicitly excluded from the trigger set.

### Error Context Propagation Pattern
All subagents in a multi-agent system use a standardized error response schema. The schema includes: the failed operation (what was attempted), the error category (network, authentication, validation, not_found, rate_limit, unknown), the error message (human-readable description), retryability (boolean plus optional retry_after delay), severity (warning, error, fatal), and suggested alternatives (other approaches the coordinator might try). The coordinator applies category-specific handling: network errors trigger retry with backoff, authentication errors trigger credential refresh or escalation, not_found results are treated as valid empty responses (not errors), rate_limit errors trigger delayed retry, and unknown errors are logged with full context and escalated. This pattern eliminates the ambiguity between access failures and valid empty results.

### External Memory (Scratchpad) Pattern
During extended exploration tasks — particularly large codebase analysis — the agent maintains a structured scratchpad file that serves as external memory. The scratchpad is organized into sections: an overall summary that is updated as understanding deepens, a findings section that records specific discoveries with file paths and line numbers, a dependencies section that maps relationships between components, an open questions section that tracks unresolved issues, and a next steps section that maintains the exploration plan. The agent writes to the scratchpad after each significant discovery and re-reads it before making strategic decisions about what to explore next. This externalizes the agent's accumulated knowledge beyond the context window's effective attention span.

### Stratified Review Pattern
Quality measurement uses stratified random sampling across all relevant categories. The review framework defines strata along two primary dimensions: document type (or task type) and field type (or output component). Each stratum is sampled independently, with sample sizes determined by the desired confidence level and the acceptable margin of error. Accuracy is measured and reported per stratum, not in aggregate. Confidence score calibration uses a labeled validation set processed through the system, producing calibration curves per stratum that map model-reported confidence to actual accuracy. Review routing thresholds are set based on these calibration curves, not on raw model confidence values.

### Provenance Chain Pattern
Every claim in a synthesized output is linked to its source through a structured mapping. Each subagent returns findings in a format that includes source metadata: source identifier (URL, document ID, database name), publication or collection date, author or organization, and a reliability indicator. The coordinator preserves these source tags during synthesis, producing output where each claim includes its source lineage. When sources conflict, the conflict is surfaced explicitly with both sources cited. The final output includes a sources section that lists all sources with their metadata, and each claim in the body references one or more sources from this section.

## Scenario Walkthrough

### Multi-Agent Research System with Error Handling and Source Attribution

Consider a system tasked with producing a market analysis report. The coordinator receives a request: "Analyze the current state of the electric vehicle battery market, including market size, key players, and recent technological developments."

**Step 1: Task Decomposition.** The coordinator decomposes the request into three subtasks and delegates each to a specialist subagent: (a) a web search subagent to find recent market data and industry reports, (b) a document analysis subagent to extract information from a set of provided industry PDFs, and (c) a synthesis subagent to combine the findings into a coherent report. Each subagent receives a focused prompt specifying what information to find and the required output format, including source attribution requirements.

**Step 2: Subagent Execution with Source Tracking.** The web search subagent executes multiple searches, retrieves relevant pages, and returns structured findings. Each finding includes the claim, the source URL, the publication date, and the organization. For example: `{ "claim": "Global EV battery market valued at $46.7B in 2023", "source_url": "https://example.com/report", "source_org": "Industry Research Group", "publication_date": "2024-03-15" }`. The document analysis subagent processes the PDFs and returns similar structured findings, with document name and page number as the source reference.

**Step 3: Error Handling.** During execution, the web search subagent encounters a timeout on one of its search queries. Instead of silently dropping that search, it returns a structured error alongside its successful results: `{ "partial_results": [...successful findings...], "errors": [{ "operation": "search_query_3", "query": "EV battery recycling market", "error_type": "connection_timeout", "retryable": true, "retry_after_seconds": 10, "message": "Search API timed out after 30s" }] }`. The coordinator sees that the battery recycling data is missing due to a temporary failure, retries after 10 seconds, and successfully retrieves the missing data.

Meanwhile, the document analysis subagent queries for a specific report that was referenced in the task but returns: `{ "status": "not_found", "operation": "document_lookup", "document": "Q4_2024_Battery_Report.pdf", "message": "Document not present in provided collection" }`. The coordinator correctly interprets this as a valid empty result — the document does not exist — rather than an access failure, and proceeds without it rather than retrying.

**Step 4: Synthesis with Provenance.** The synthesis subagent receives the combined findings from both source subagents, each tagged with source metadata. It produces a report where each section cites its sources. When it encounters conflicting data — the web search found a market size of $46.7B from one source and $44.1B from another — it surfaces the conflict: "Market size estimates vary: Industry Research Group reports $46.7B (March 2024 data), while Market Analytics Corp reports $44.1B (January 2024 data). The discrepancy may reflect different market scope definitions." The temporal metadata is included so the reader can assess data freshness.

**Step 5: Final Output.** The coordinator delivers the report to the user with a completeness note: all three research areas (market size, key players, technology developments) were successfully covered. One data source (the Q4 report) was unavailable but was not critical. All claims in the report include source attribution, and two source conflicts were surfaced for the reader's judgment.

This walkthrough demonstrates the integration of error propagation (structured errors enabling retry and graceful degradation), provenance tracking (source metadata preserved through synthesis), and conflict surfacing (disagreements between sources made explicit rather than hidden).

## Practice Exercises

### Exercise 1: Context Management for Long Customer Support Conversations

Design a context management strategy for a customer support agent that handles conversations averaging 40 turns. The agent discusses order details (order numbers, product names, prices, quantities, dates), applies discount codes, processes returns, and coordinates shipping changes. Your design should address:

- Which data points must be pinned and preserved through any summarization
- How the pinned facts block is structured and positioned in the context
- At what conversation length summarization should begin, and how it is applied
- How tool results (order lookups, inventory checks, shipping queries) are managed to prevent context bloat
- How the lost-in-the-middle effect is mitigated for mid-conversation commitments

A strong answer will include a concrete pinned facts schema (listing the specific fields to extract), a summarization trigger and strategy, and a tool result management approach that limits context accumulation.

### Exercise 2: Structured Error Responses for a Multi-Agent System

Design the error response schema for a multi-agent system with three subagents: a data retrieval subagent (queries databases and APIs), a document processing subagent (reads and extracts from PDFs and web pages), and a calculation subagent (performs numerical analysis on extracted data). For each subagent, define:

- The error categories it can encounter (at least 4 per subagent)
- The structured error response format with all required fields
- How the coordinator distinguishes access failures from valid empty results for each subagent
- The coordinator's error handling logic: which errors trigger retry, which trigger fallback, which trigger escalation, and which trigger graceful degradation

A strong answer will demonstrate that the error schema is standardized across subagents while allowing subagent-specific error categories, and that the coordinator's handling logic is deterministic rather than model-judged.

### Exercise 3: Human Review Workflow with Stratified Sampling

Design a human review workflow for an automated system that processes four document types: invoices, purchase orders, contracts, and receipts. The system extracts five fields from each: vendor name, date, line items, total amount, and payment terms. Your design should address:

- The stratified sampling strategy: how many documents to sample from each type, and why
- Per-type and per-field accuracy measurement: how results are structured and reported
- Confidence calibration: how the labeled validation set is constructed and how calibration curves are produced
- Review routing: how the calibrated confidence thresholds are used to decide which extractions go to human review
- Detection of the aggregate masking problem: how the workflow ensures that poor accuracy on a specific type or field is surfaced

A strong answer will specify concrete sample sizes with statistical justification, show how aggregate metrics are decomposed, and describe the calibration process with enough detail to implement it.

## Exam Tips

- Despite being weighted at only 15%, Domain 5 concepts appear in 4 of the 6 exam scenarios — making it one of the most cross-cutting domains. Do not allocate study time purely proportional to weight.
- "Sentiment-based escalation" and "self-reported confidence scores" are almost always wrong answers when presented as escalation triggers or quality mechanisms. The exam tests whether you understand why these are unreliable, not just that they are.
- The distinction between access failures and valid empty results is a core testing concept. Any time a question presents an error handling scenario, check whether the answer choices confuse these two cases.
- Scratchpad files and subagent delegation are the two key strategies for large codebase exploration. If a question asks about managing context during extended code analysis, the correct answer involves one or both of these.
- Aggregate accuracy metrics masking per-type or per-field problems is the central concept in human review workflow questions. If a question describes a system that reports high overall accuracy, the correct concern is almost always about what the aggregate is hiding.
- Source attribution loss during summarization is heavily tested. Questions will present multi-source synthesis scenarios where the correct design preserves claim-source mappings and surfaces conflicts between sources.
- Context pinning — extracting critical values before summarization — is the correct answer for long conversation management, not aggressive summarization alone.
- When a question presents a multi-agent error scenario, look for answers that include structured error context (what failed, why, retryable, alternatives). Generic error statuses are always the wrong pattern.
