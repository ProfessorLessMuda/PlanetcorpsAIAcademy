# Module 01 - Prompt Waste & Lean AI Principles

## Exam Weighting: 15%

## Objective
Identify and eliminate the eight types of prompt waste by applying Lean manufacturing principles to AI interaction design. Understand HALO (High-Alignment Logical Output) as the target state for every AI interaction.

## Task Statements Covered
- 1.1: Identify the 8 types of prompt waste in AI interactions
- 1.2: Map Lean manufacturing principles to AI prompt workflows
- 1.3: Diagnose waste in existing prompts using before/after analysis
- 1.4: Apply a continuous improvement mindset to prompt iteration

## Key Concepts

### 1.1 The Eight Types of Prompt Waste

Lean manufacturing identifies eight categories of waste (known by the acronym DOWNTIME in some frameworks, or the Toyota Production System's original seven plus one). Each maps directly to a failure mode in how people interact with AI models. Understanding these categories is the foundation of Lean AI Sigma because you cannot eliminate waste you cannot name.

**Overproduction** is generating far more output than the task requires. This happens when a prompt lacks clear constraints on length, scope, or deliverable format. A user who asks "tell me everything about marketing" and receives 2,000 words when they needed a 150-word summary has triggered overproduction. The model did exactly what was asked, but what was asked was wasteful. Overproduction consumes tokens on both the output side (generation cost) and the downstream side (the user must read and filter irrelevant content). In Lean terms, overproduction is considered the worst form of waste because it triggers other wastes -- the user must now spend time reading unnecessary content (motion waste), may need to re-prompt for a focused version (rework), and the extra tokens cost money (resource waste).

**Waiting** manifests as idle time between prompt iterations. When a user sends a vague prompt, reads the output, realizes it missed the mark, rephrases, and tries again, each gap between attempts is waiting waste. In a manufacturing context, waiting is a machine sitting idle while materials are prepared. In AI interactions, it is the human sitting idle while they figure out how to rephrase a failed prompt. The root cause is almost always insufficient upfront thinking -- the user did not invest time designing the prompt before executing it. Lean AI Sigma's core principle ("think before prompting") directly targets this waste type.

**Unnecessary Tokens** are the AI equivalent of unnecessary inventory -- tokens that exist in the prompt but add no value to the output. Redundant instructions ("Please make sure to definitely include..."), filler phrases ("I was wondering if you could possibly..."), and repeated context that the model already has in its window are all unnecessary tokens. Every token has a cost: financial (API pricing is per-token), latency (more tokens take longer to process), and environmental (compute cycles require energy). Unnecessary tokens are insidious because they feel harmless individually but compound at scale. An organization sending 10,000 prompts per day with 50 unnecessary tokens each is burning 500,000 tokens daily for zero value.

**Over-processing** occurs when the prompt asks the model to perform work beyond what the task requires. Requesting "a comprehensive analysis with executive summary, detailed breakdown, recommendations, and implementation timeline" when only the recommendations are needed is over-processing. The model diligently produces all four sections, consuming tokens and time to generate content that will never be read. Over-processing is closely related to overproduction but is subtly different: overproduction is too much of what you asked for, while over-processing is asking for more work than the task warrants.

**Excess Context Inventory** is the prompt equivalent of excess inventory on a factory floor. It occurs when a user loads a prompt with background information, reference material, or historical context that the model does not need for the specific task. While context is essential for good outputs, too much context is wasteful: it consumes input tokens, can distract the model from the core task (diluting attention across irrelevant information), and may push important instructions further from the edges of the context window where attention is strongest. The goal is just-in-time context -- providing exactly the information the model needs, when it needs it, and nothing more.

**Unnecessary Motion** in AI interactions is the user repeatedly rephrasing, reformatting, or restructuring the same request. A user who sends the same core request five different ways, each time adjusting wording because outputs are inconsistent, is experiencing motion waste. The root cause is typically a poorly structured prompt that leaves too much to interpretation. Each rephrasing attempt costs tokens, time, and cognitive effort without addressing the structural problem. CORPS (covered in Module 02) eliminates unnecessary motion by providing a repeatable structure that removes the guesswork from prompt construction.

**Defects** are outputs that fail to meet the stated or implied objective. A defect might be factually incorrect content, wrong formatting, missing sections, misunderstood tone, or any output that requires correction or rework. Defects are the most visible form of waste because the user immediately recognizes the output is wrong. However, the true cost of defects extends beyond the flawed output itself: each defect triggers rework (a new prompt to fix the problem), additional tokens (both the correction prompt and the regenerated output), and lost time. In Lean manufacturing, defect prevention is always cheaper than defect correction. The same principle applies to prompting: investing time in a precise, well-structured prompt prevents defects far more effectively than iterating after failure.

**Underutilized Talent** is the failure to leverage the model's full capabilities. This manifests as users who treat AI as a simple question-answer machine when it can handle complex analysis, multi-step reasoning, structured output generation, and creative problem-solving. Over-constraining the model with excessively prescriptive instructions can also trigger this waste: telling the model exactly what to write word-by-word defeats the purpose of using an AI. The balance is providing enough structure to prevent defects while leaving enough latitude for the model to apply its strengths.

### 1.2 Lean Principles Applied to AI

The core philosophy of Lean is simple: maximize value while minimizing waste. In manufacturing, value is defined as any activity the customer is willing to pay for. In AI interactions, value is any token that contributes to achieving the stated objective. Everything else is waste.

**Value Stream Mapping** is the Lean practice of documenting every step in a process to identify which steps add value and which are waste. Applied to AI prompting, value stream mapping means tracing the full lifecycle of an interaction: the time spent thinking about the task, the tokens spent on the prompt, the tokens generated in response, the time spent evaluating the output, and any subsequent iterations. Each element can be classified as value-adding (directly contributes to the objective), non-value-adding but necessary (setup, formatting instructions), or pure waste (redundant instructions, failed iterations). Mapping this stream reveals where the biggest waste concentrations are and where improvement efforts should focus.

**Pull-Based Interaction Design** reverses the default AI interaction pattern. Most users operate in push mode: they push a prompt to the model and hope the output matches their needs. Pull-based design starts from the desired output and works backward to determine what prompt and context are necessary to produce it. This is the thinking-before-prompting principle in action. Instead of "let me write a prompt and see what happens," pull-based design asks "what do I need the output to look like, and what is the minimum input required to produce it?" This approach naturally eliminates overproduction (the output scope is defined upfront), defects (the success criteria are explicit), and unnecessary tokens (only the required input is included).

**Just-in-Time Context Delivery** borrows from Lean's just-in-time inventory management. Instead of front-loading every possible piece of context into the initial prompt, JIT context delivers information only when the model needs it for the current step. In multi-turn interactions, this means providing background context in the first turn, task-specific details in the second turn, and refinement criteria in the third -- rather than cramming everything into a single massive prompt. JIT context reduces token waste, keeps each turn focused, and avoids the "lost in the middle" problem where critical information buried in a large context block receives less model attention.

**Single-Piece Flow** in manufacturing means processing one item at a time through each step rather than batching. Applied to prompting, it means giving the model one clear objective per prompt rather than bundling multiple unrelated tasks. A prompt that asks "write a blog post, create three social media summaries, and draft an email to the editor" is a batch. Each task has different requirements, and bundling them increases the probability of defects in at least one output. Single-piece flow produces higher-quality results because the model can devote its full attention to one objective at a time.

**Kaizen** (continuous improvement) is the practice of making small, incremental improvements continuously rather than large improvements occasionally. Applied to prompting, Kaizen means treating every AI interaction as a learning opportunity. Did the prompt produce the desired output on the first try? If not, what specifically caused the failure? Was it missing context, an unclear objective, or an inappropriate role assignment? Each failure diagnosis improves the next prompt. Over time, Kaizen transforms an organization's prompting from trial-and-error to precision engineering. The PDCA cycle (Module 05) is the operational framework for practicing Kaizen.

### 1.3 Waste Identification Techniques

Diagnosing waste requires a structured methodology, not gut feeling. The before/after prompt comparison is the primary diagnostic tool in Lean AI Sigma.

**Before/After Analysis** works as follows. Take an existing prompt (the "before" state) that produces acceptable but potentially wasteful results. Document: the prompt text, its token count, the number of iterations required to get an acceptable output, the output token count, and the quality of the final output. Then redesign the prompt using CORPS principles (Module 02) to create the "after" state. Document the same metrics. The difference between before and after quantifies the waste that existed in the original prompt. A well-executed before/after analysis typically reveals 30-50% token reduction while maintaining or improving output quality.

**Token Counting as Waste Measurement** provides an objective metric for waste. Unlike subjective assessments ("this prompt feels too long"), token counts are precise and comparable. Establishing a tokens-per-successful-output baseline for common tasks creates a benchmark against which improvements can be measured. If your standard customer email prompt uses 200 input tokens and produces 300 output tokens across 2 iterations (total: 200 + 200 + 300 + 300 = 1,000 tokens for a successful output), reducing iterations to 1 through better prompting drops the total to 500 tokens -- a 50% improvement.

**The Five Whys Applied to Prompt Failures** is a root cause analysis technique borrowed from Toyota's production system. When a prompt produces a defective output, ask "why" five times to trace the failure to its root cause. Why did the output miss the target? Because the model focused on the wrong aspect. Why did it focus on the wrong aspect? Because the prompt did not specify the priority. Why did the prompt not specify the priority? Because the user assumed the model would infer it. Why did the user assume inference? Because the user treats the model like a human colleague who shares context. Why does the user treat the model this way? Because they have not been trained in prompt design. The root cause is not the model's failure -- it is a gap in the user's prompting methodology, which CORPS directly addresses.

### 1.4 HALO: The Target State

HALO -- High-Alignment Logical Output -- is the target state for every AI interaction in Lean AI Sigma. A HALO output is one where the model's response is precisely aligned to the stated objective, logically structured, and produced with minimal waste.

Achieving HALO is not about getting the "perfect" output. It is about getting the right output on the first pass, with no unnecessary tokens consumed and no rework required. HALO is the Lean AI equivalent of "right first time" from quality management. The gap between a typical first-attempt output and a HALO output represents the waste opportunity in that interaction.

HALO connects to every other concept in this program. The CORPS framework (Module 02) is the tool for designing prompts that produce HALO outputs. Context engineering (Module 03) ensures the model has what it needs -- and nothing more -- to achieve alignment. Token efficiency (Module 04) minimizes the resource cost of achieving HALO. And the LAI score (Module 05) measures how close each interaction comes to the HALO ideal. The entire Green Belt curriculum is, fundamentally, a methodology for consistently achieving HALO across all AI interactions.
